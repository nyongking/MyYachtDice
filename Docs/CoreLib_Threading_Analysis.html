SECTION 01

현재 CoreLib 스레딩 시스템 요약

구성요소 상태 비고
ThreadManager ✅ 완료 워커 스레드 관리, 글로벌 큐 분배
Lock (RW Spinlock) ✅ 완료 재진입 가능, 10초 타임아웃, 데드락 감지
JobQueue / ConcurrentJobQueue ✅ 완료 비동기 잡 실행, 시간 슬라이싱
MainThreadQueue ✅ 완료 GPU/메인 스레드 전용 지연 작업
JobTimer ✅ 완료 지연 예약 잡 (weak_ptr 안전)
DeadLockProfiler ✅ 완료 DEBUG 전용 DFS 순환 감지
LockQueue<T> ✅ 완료 범용 스레드 안전 큐
CoreTLS ✅ 완료 TLS 4개 변수

현재 시스템은 "잡을 큐에 넣고 워커가 처리"하는 fire-and-forget 모델입니다. 게임 엔진으로서 실질적으로 부족한 영역을 우선순위별로 정리했습니다.

SECTION 03

추가 계획

Phase 1 — 즉시 필요 (높은 실용성)
Phase 1
1-1

Future<T> / Promise<T> — 비동기 결과 반환

현재 Job은 결과를 돌려받을 방법이 없습니다. 리소스 비동기 로딩에서 "로드 완료 시 결과를 받아 처리"하는 패턴이 필요합니다. 

// 사용 예시
auto future = jobQueue->DoAsyncWithResult<Texture*>([](){
    return LoadTextureFromDisk("path.dds");
});

// 나중에
if (future.IsReady()) {
    Texture* tex = future.Get();
}
  • SharedState<T> — 결과 + ready 플래그 + continuation
  • Promise<T> — 생산자 측 (SetValue / SetException)
  • Future<T> — 소비자 측 (IsReady / Get / Then)
Phase 1
1-2

ParallelFor — 병렬 배치 처리

물리, 파티클, AI 등에서 대량 오브젝트를 병렬 처리할 유틸리티가 없습니다. 

// 사용 예시
ParallelFor(0, rigidBodies.size(), [&](int32 i) {
    rigidBodies[i].IntegrateForces(dt);
});
  • 인덱스 범위를 워커 수에 따라 청크 분할
  • 내부적으로 JobQueue + 카운터 기반 완료 대기
Phase 1
1-3

CountdownEvent (Barrier / Latch)

여러 워커 작업의 완료를 동기적으로 대기하는 메커니즘이 없습니다. 

CountdownEvent event(taskCount);
for (auto& task : tasks) {
    jobQueue->DoAsync([&]() {
        task.Process();
        event.Signal();
    });
}
event.Wait(); // 모든 태스크 완료까지 대기
  • Atomic<int32> 카운터 + Windows Event 기반 대기
  • 순수 스핀 대기 회피 → CPU 효율 개선
Phase 2 — 아키텍처 강화
Phase 2
2-1

TaskGraph — 잡 의존성 그래프

현재 잡 간 순서 보장이 불가합니다. "A, B 완료 후 C 실행" 같은 의존성 표현이 필요합니다. 

auto taskA = graph.AddTask("PhysicsBroadPhase",  [](){ /* ... */ });
auto taskB = graph.AddTask("PhysicsNarrowPhase", [](){ /* ... */ });
auto taskC = graph.AddTask("PhysicsResolve",     [](){ /* ... */ });

taskC->DependsOn(taskA);
taskC->DependsOn(taskB);
graph.Execute(); // A, B 병렬 → C 순차
  • 물리 파이프라인 (BroadPhase → NarrowPhase → Resolve) 즉시 활용
  • DAG 기반 토폴로지 정렬 + 병렬 실행
Phase 2
2-2

SpinWait + EventSignal — 하이브리드 대기

현재 Lock은 순수 스핀락입니다. 경합이 긴 경우 CPU를 낭비합니다. 

// 짧은 경합: 스핀 → 긴 경합: OS 이벤트 대기로 전환
class SpinWait {
    void SpinOnce();
    // YieldProcessor → SwitchToThread → Sleep(1) 점진 전환
    bool NextSpinWillYield();
};
  • YieldProcessor → SwitchToThread → Sleep(1) 점진 전환
  • 경합 강도에 따른 자동 전략 전환
Phase 2
2-3

ThreadSafeHashMap<K, V> — 스레드 안전 해시맵

리소스 매니저(ResourceCache<T>)가 현재 Lock + unordered_map을 수동으로 감싸고 있습니다. 범용 concurrent 컨테이너가 있으면 코드 중복을 줄일 수 있습니다.

  • 분리 잠금 (Striped Locking) 또는 CAS 기반 구현
  • ResourceCache<T> 코드 단순화
  • GeometryManager / ShaderManager / TextureManager 즉시 활용
Phase 3 — 고급 최적화 (추후)
기능 설명 용도
