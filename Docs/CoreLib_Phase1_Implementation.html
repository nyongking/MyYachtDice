SECTION 02

CountdownEvent

2.1 클래스 개요

헤더 전용. Windows Event 기반의 다중 작업 완료 대기 동기화 프리미티브입니다.

핵심 설계 원칙:

  • ▸ Atomic<int32> 카운터로 잔여 작업 수 추적
  • ▸ Windows CreateEvent (manual-reset) 기반 OS 레벨 대기 — CPU를 양보하므로 스핀락보다 효율적
  • ▸ RAII: 소멸자에서 CloseHandle 자동 호출
  • ▸ 복사/이동 금지
2.2 멤버 변수
멤버 타입 설명
m_count Atomic<int32> 잔여 작업 카운터. Signal()마다 1 감소
m_event HANDLE Windows manual-reset Event. 카운터 0 도달 시 신호
2.3 API 상세
CountdownEvent(int32 count) — 생성자
  • count로 카운터 초기화
  • count ≤ 0이면 처음부터 신호 상태(signaled)로 Event 생성
  • ASSERT_TRIG_CRASH로 Event 생성 실패 시 크래시
~CountdownEvent() — 소멸자
  • CloseHandle(m_event) 호출. RAII 패턴.
void Signal()
  • m_count.fetch_sub(1)로 원자적 감소
  • 이전 값이 1이었으면 (즉 0으로 전환) SetEvent(m_event) 호출 → 대기 중인 스레드 깨움
  • 스레드 안전: lock-free (atomic 연산만 사용)
void Wait()
  • m_count.load() <= 0이면 즉시 반환 (이미 완료)
  • 그 외 WaitForSingleObject(m_event, INFINITE)로 블로킹 대기
  • OS 레벨 대기이므로 CPU 점유 없음
bool Wait(uint32 timeoutMs)
  • 타임아웃(밀리초) 지정 대기
  • 반환값: true = 완료 (WAIT_OBJECT_0), false = 타임아웃
bool IsComplete() const
  • 논블로킹 완료 확인
  • m_count.load() <= 0 반환
void Reset(int32 count)
  • 이벤트 재사용. 새 count로 카운터 재설정
  • count > 0: ResetEvent (비신호 상태)
  • count ≤ 0: SetEvent (신호 상태)
2.4 사용 예시 
// 5개 비동기 작업 완료 대기
Core::CountdownEvent event(5);

for (int i = 0; i < 5; ++i)
{
    jobQueue->DoAsync(true, [&event]()
    {
        DoHeavyWork();
        event.Signal();  // 완료 → 카운터 -1
    });
}

event.Wait();  // 카운터가 0이 될 때까지 블로킹 대기
// 여기 도달하면 5개 작업 모두 완료

// 재사용
event.Reset(3);
// ... 3개 추가 작업 제출 ...
event.Wait();

CountdownEvent 동작 시퀀스 다이어그램
Thread A (호출자)           Worker 1          Worker 2          Worker 3
     │                        │                 │                 │
     ├─ new CountdownEvent(3) │                 │                 │
     │   m_count = 3          │                 │                 │
     │   m_event = 비신호     │                 │                 │
     │                        │                 │                 │
     ├─ DoAsync(job1) ────────→│                 │                 │
     ├─ DoAsync(job2) ─────────┼────────────────→│                 │

     ├─ Wait()                │                 │                 │

SECTION 03

Future<T> / Promise<T>

3.1 아키텍처 개요

3개 클래스 + 1개 팩토리 함수로 구성된 비동기 결과 반환 시스템입니다.

클래스 관계도
MakePromise<T>() ──→ SharedState<T> (shared_ptr)
                        ↗                    ↘
               Promise<T>               Future<T>
              (생산자 측)              (소비자 측)
              SetValue()               Get()
              SetException()           TryGet()
                                         Then()
                                         IsReady()
3.2 SharedState<T> — 내부 공유 상태

Promise와 Future가 shared_ptr로 공유하는 내부 상태 구조체입니다.

멤버 변수

멤버 타입 설명
m_ready Atomic<bool> 결과 설정 완료 플래그
m_value T 결과 값 저장
m_exception std::exception_ptr 예외 저장 (실패 시)
m_continuation CallbackType Then()으로 등록된 연속 콜백
m_event HANDLE Windows Event. Get() 블로킹 대기용
m_locks[1] Lock USE_LOCK 매크로. continuation 등록 보호

API 상세

void SetValue(T value) — 생산자 측에서 결과 설정
  • m_ready가 이미 true이면 ASSERT_TRIG_CRASH (이중 설정 방지)
  • m_value에 move 저장 → m_ready = true → SetEvent → continuation 콜백 실행(있으면)
  • continuation 접근은 WRITE_LOCK으로 보호
void SetException(std::exception_ptr ex) — 예외 설정
  • SetValue와 동일 흐름, 단 m_exception에 저장
  • 소비자가 Get() 시 std::rethrow_exception으로 재전파
T& Get() — 블로킹 대기 후 결과 반환
  • m_ready가 false면 WaitForSingleObject(INFINITE) 대기
  • 예외가 설정되어 있으면 rethrow_exception
  • 값 참조 반환
bool TryGet(T& out) — 논블로킹 시도
  • m_ready false → return false (즉시)
  • 예외 있으면 rethrow
  • 값을 out에 복사 후 return true
void SetContinuation(CallbackType cb) — 연속 콜백 등록
  • WRITE_LOCK 내에서 ready 상태 확인
  • 아직 미완료 → 콜백 저장 (나중에 SetValue/SetException에서 실행)
  • 이미 완료 → 콜백 즉시 실행
  • 경쟁 조건 방지: Lock으로 "ready 확인 + 콜백 저장"을 원자적으로 수행
bool IsReady() const
  • m_ready.load() 반환
3.3 Future<T> — 소비자 측
메서드 반환 설명
Future() — 기본 생성자 (빈 상태)
Future(Ref<SharedState<T>>) — SharedState 연결
bool IsReady() bool 결과 준비 여부. ASSERT: state 유효
T& Get() T& 블로킹 대기 → 결과 참조 반환. 예외 시 rethrow
bool TryGet(T& out) bool 논블로킹 시도. true = 값 복사됨
void Then(CallbackType) void 완료 시 실행할 콜백 등록
bool IsValid() bool SharedState가 null이 아닌지
3.4 Promise<T> — 생산자 측
메서드 반환 설명
Promise() — 기본 생성자 (빈 상태)
Promise(Ref<SharedState<T>>) — SharedState 연결
void SetValue(T) void 결과 설정. ASSERT: state 유효. 1회만 호출 가능
void SetException(exception_ptr) void 예외 설정. ASSERT: state 유효. 1회만 호출 가능
bool IsValid() bool SharedState가 null이 아닌지
3.5 MakePromise<T>() — 팩토리 함수 
template<typename T>
std::pair<Promise<T>, Future<T>> MakePromise()
{
    auto state = std::make_shared<SharedState<T>>();
    return { Promise<T>(state), Future<T>(state) };
}

SharedState<T>를 생성하고 Promise/Future 쌍으로 반환합니다. 구조적 바인딩 사용 가능: auto [promise, future] = MakePromise<int>();

3.6 사용 예시 
// 기본 사용
auto [promise, future] = Core::MakePromise<int>();

jobQueue->DoAsync(true, [p = std::move(promise)]() mutable
{
    int result = HeavyComputation();
    p.SetValue(result);
});

// 나중에 결과 확인
if (future.IsReady())
{
    int value = future.Get();
}

// 또는 블로킹 대기
int value = future.Get();  // 완료될 때까지 대기

// 논블로킹 폴링
int value;
if (future.TryGet(value))
    UseResult(value);

// 콜백 체이닝
future.Then([&]()
{
    // future가 완료되면 자동 실행
    ProcessResult(future.Get());
});

// 예외 전파
auto [promise2, future2] = Core::MakePromise<Texture*>();
jobQueue->DoAsync(true, [p = std::move(promise2)]() mutable
{
    try {
        p.SetValue(LoadTexture("path.dds"));
    } catch (...) {
        p.SetException(std::current_exception());
    }
});

try {
    Texture* tex = future2.Get();
} catch (const std::exception& e) {
    // 워커 스레드의 예외가 여기서 재전파
}
SECTION 04

ParallelFor

4.1 함수 시그니처 
// 자동 청크 크기 (하드웨어 코어 수 기반)
void ParallelFor(int32 start, int32 end, std::function<void(int32)> body);

// 수동 청크 크기 지정
void ParallelFor(int32 start, int32 end, int32 chunkSize, std::function<void(int32)> body);
4.2 내부 동작

자동 청크 크기 오버로드

  1. 1 totalCount = end - start 계산. ≤ 0이면 즉시 반환
  2. 2 std::thread::hardware_concurrency()로 CPU 코어 수 획득
  3. 3 chunkSize = (totalCount + workerCount - 1) / workerCount (올림 나눗셈)
  4. 4 수동 오버로드 호출

수동 청크 크기 오버로드

  1. 1 totalCount ≤ 0 → 즉시 반환
  2. 2 chunkSize ≤ 0 → 1로 보정
  3. 3 totalCount ≤ chunkSize → 인라인 실행 (스레드 오버헤드 제거). 단순 for 루프
  4. 4 chunkCount = (totalCount + chunkSize - 1) / chunkSize 계산
  5. 5 CountdownEvent countdown(chunkCount) 스택에 생성
  6. 6
    각 청크마다:
    • – chunkStart, chunkEnd 계산 (마지막 청크는 end로 클램프)
    • – shared_ptr<JobQueue> 생성
    • – DoAsync(pushOnly=true, ...) 호출 — 내부적으로 PushGlobalQueue 자동 호출
    • – 람다: for (i = chunkStart; i < chunkEnd; ++i) body(i); 후 countdown.Signal()
  7. 7 countdown.Wait() — 모든 청크 완료까지 블로킹 대기
4.3 성능 특성
항목 설명
인라인 임계값 totalCount ≤ chunkSize일 때 직접 실행 (오버헤드 0)
청크 분할 CPU 코어 수 기반 자동 분할
대기 방식 CountdownEvent (OS Event) — CPU 양보
잡 제출 청크당 1개 JobQueue, PushGlobalQueue로 워커 풀에 분배
메모리 스택 기반 CountdownEvent + 힙 기반 JobQueue (shared_ptr)
4.4 청크 분할 시각화

ParallelFor(0, 100, body) — 코어 4개 가정, chunkSize = 25

Chunk 0
[0, 25)
Worker A
↓ Signal()
Chunk 1
[25, 50)
Worker B
↓ Signal()
Chunk 2
[50, 75)
Worker C
↓ Signal()
Chunk 3
[75, 100)
Worker D
↓ Signal()
⬇ CountdownEvent(4) Wait() → 4번의 Signal() 완료 후 반환
4.5 사용 예시 
// 기본: 자동 청크 분할
std::vector<RigidBody> bodies(1000);
Core::ParallelFor(0, (int32)bodies.size(), [&](int32 i)
{
    bodies[i].IntegrateForces(dt);
});

// 수동 청크 크기
Core::ParallelFor(0, (int32)particles.size(), 64, [&](int32 i)
{
    particles[i].Update(dt);
});

// 작은 범위 → 자동 인라인 (스레드 오버헤드 없음)
Core::ParallelFor(0, 3, [&](int32 i)
{
    // 3개면 청크 분할 없이 직접 실행
    smallArray[i].Process();
});
SECTION 05

JobQueue 확장 — DoAsyncWithResult<T>()

5.1 추가된 메서드 
template<typename T>
Core::Future<T> DoAsyncWithResult(bool pushOnly, std::function<T()> callback)

JobQueue에 추가된 새 메서드. 기존 DoAsync()와 동일하게 비동기 실행하되, 결과를 Future<T>로 반환합니다.

5.2 내부 동작
  1. 1 MakePromise<T>()로 Promise/Future 쌍 생성
  2. 2 Promise를 캡처한 람다를 Job으로 래핑
  3. 3 람다 내부: try { promise.SetValue(callback()); } catch (...) { promise.SetException(current_exception()); }
  4. 4 Push(job, pushOnly) — 기존 JobQueue 경로로 제출
  5. 5 Future 반환
5.3 사용 예시 
auto future = jobQueue->DoAsyncWithResult<Texture*>(true, []()
{
    return LoadTextureFromDisk("albedo.dds");
});

// 메인 루프에서 폴링
Texture* tex = nullptr;
if (future.TryGet(tex))
{
    material->SetTexture("AlbedoMap", tex);
}

SECTION 07

주의사항 및 제약

⚠ 주의 1 — CountdownEvent Signal 과다 호출 금지
count보다 많이 Signal()하면 카운터가 음수로 진입합니다. 정확히 count번만 호출해야 합니다.
⚠ 주의 2 — Promise는 1회만 설정 가능
SetValue() 또는 SetException()을 2번 호출하면 ASSERT_TRIG_CRASH합니다. 한 Promise에 한 번만 결과를 설정하세요.
⚠ 주의 3 — Future::Get()은 블로킹
메인 스레드에서 호출 시 프레임 멈춤이 발생할 수 있습니다. 메인 스레드에서는 TryGet() 또는 Then()을 권장합니다.
⚠ 주의 4 — ParallelFor는 동기 함수
내부에서 Wait()로 모든 청크 완료까지 대기합니다. 호출 스레드는 완료까지 블로킹됩니다.
⚠ 주의 5 — ParallelFor 람다에서 공유 자원 접근 주의
body 내부에서 같은 컨테이너의 다른 인덱스를 접근하는 것은 안전하지만, 공유 변수 쓰기에는 별도 동기화가 필요합니다.
