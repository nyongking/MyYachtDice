SECTION 02

SpinWait

2.1 클래스 개요

헤더 전용. 반복 횟수에 따라 자동으로 대기 전략을 전환하는 점진적 백오프 유틸리티입니다.

설계 원칙:

  • ▸ 짧은 경합: CPU에 가깝게 유지 (YieldProcessor)
  • ▸ 중간 경합: 다른 스레드에 양보 (SwitchToThread)
  • ▸ 긴 경합: OS 타임슬라이스 양보 (Sleep)
  • ▸ 기존 Lock의 고정 MAX_SPIN_COUNT=5000 스핀락보다 효율적
2.2 백오프 단계
Phase 반복 횟수 함수 동작 CPU 비용
1 0 ~ 9 YieldProcessor() CPU 파이프라인 힌트, 동일 코어 유지 높음 (busy-wait)
2 10 ~ 19 SwitchToThread() 동일 우선순위 스레드에 실행 양보 중간
3 20+ Sleep(1) OS 타임슬라이스 양보, 다른 코어/스레드 허용 낮음
2.3 멤버 변수
멤버 타입 설명
m_count int32 현재 SpinOnce() 호출 횟수
YIELD_THRESHOLD static constexpr int32 = 10 Phase 2 전환 임계값
SLEEP_THRESHOLD static constexpr int32 = 20 Phase 3 전환 임계값
2.4 API 상세
void SpinOnce()
  • m_count에 따라 3단계 백오프 자동 전환
  • 호출 후 m_count 증가
  • 스레드 안전: 인스턴스별 사용 (TLS 또는 로컬 변수), 공유 불필요
bool NextSpinWillYield() const
  • 다음 SpinOnce()가 OS 양보(Phase 2+)를 수행하는지 확인
  • m_count >= YIELD_THRESHOLD 반환
void Reset()
  • m_count를 0으로 초기화. 재사용 시 호출
int32 Count() const
  • 현재 반복 횟수 반환
2.5 사용 예시 
// 리소스 준비 대기
SpinWait wait;
while (!resource.IsReady())
{
    wait.SpinOnce();
    // 자동 전환: CPU힌트(0~9) → OS양보(10~19) → Sleep(20+)
}

// Lock 내부 최적화 예시
SpinWait wait;
while (!TryAcquireLock())
{
    if (wait.NextSpinWillYield())
    {
        // 경합이 길다 → 다른 전략 고려
        break;
    }
    wait.SpinOnce();
}
2.6 백오프 전략 시각화
SpinOnce() 백오프 흐름
SpinOnce() 호출
     │
     ├─ m_count < 10 ──→ YieldProcessor()  ──→ CPU 파이프라인 힌트
     │                                          (나노초 단위, 코어 유지)
     │
     ├─ m_count < 20 ──→ SwitchToThread()  ──→ 같은 우선순위 스레드 양보
     │                                          (마이크로초 단위)
     │
     └─ m_count ≥ 20 ──→ Sleep(1)          ──→ OS 타임슬라이스 양보
                                                (밀리초 단위, CPU 해방)
     m_count++
SECTION 03

EventSignal

3.1 클래스 개요

헤더 전용. Windows auto-reset Event 기반의 단순 신호 메커니즘입니다.

CountdownEvent와의 차이:

특성 EventSignal CountdownEvent
Event 타입 Auto-reset Manual-reset
카운터 없음 Atomic<int32>
대기자 깨우기 1명만 모든 대기자
신호 후 상태 자동 비신호로 복귀 신호 유지
용도 1:1 신호 전달 N개 작업 완료 대기
3.2 멤버 변수
멤버 타입 설명
m_event HANDLE Windows auto-reset Event 핸들
3.3 API 상세
EventSignal() — 생성자
  • CreateEvent(nullptr, FALSE, FALSE, nullptr) — auto-reset, 비신호 상태로 시작
  • ASSERT_TRIG_CRASH로 실패 시 크래시
~EventSignal() — 소멸자
  • CloseHandle(m_event) 호출. RAII 패턴.
void Set()
  • SetEvent(m_event) 호출
  • 대기 중인 스레드 1명만 깨움 (auto-reset이므로 자동 비신호 복귀)
void Wait()
  • WaitForSingleObject(m_event, INFINITE) — 신호까지 블로킹 대기
bool Wait(uint32 timeoutMs)
  • 타임아웃 지정 대기
  • 반환: true = 신호 수신, false = 타임아웃
void Reset()
  • 수동으로 비신호 상태 복귀 (auto-reset이므로 보통 불필요)
3.4 사용 예시 
// 워커 → 메인 스레드 신호
EventSignal loadComplete;

// 워커 스레드
jobQueue->DoAsync(true, [&loadComplete]()
{
    LoadHeavyResource();
    loadComplete.Set();  // 완료 신호 → 대기자 1명 깨움
});

// 메인 스레드
loadComplete.Wait();  // 로드 완료까지 블로킹
UseResource();

// 타임아웃 사용
if (loadComplete.Wait(5000))
    UseResource();
else
    HandleTimeout();
SECTION 04

TaskGraph

4.1 아키텍처 개요

2개 클래스로 구성된 DAG 기반 작업 의존성 시스템입니다. 의존 관계를 선언하면 최대 병렬도를 유지하며 순서대로 실행합니다.

TaskGraph 구조 및 실행 흐름
TaskGraph
  │
  ├── AddTask("A", callback) ──→ TaskNode A (id=0, deps=0)
  ├── AddTask("B", callback) ──→ TaskNode B (id=1, deps=0)
  └── AddTask("C", callback) ──→ TaskNode C (id=2, deps=2)
                                       │
                                  C->DependsOn(A)
                                  C->DependsOn(B)
                                       │
                              Execute() 실행 시:
                              A,B 병렬 실행 → C 순차 실행
4.2 TaskNode — 내부 작업 노드

생성 방법: TaskGraph::AddTask()로만 생성 (생성자 private, friend class TaskGraph)

멤버 변수

멤버 타입 접근 설명
m_id int32 private 고유 노드 ID (자동 할당)
m_name std::string private 디버그용 이름
m_callback CallbackType private 실행할 작업
m_dependencies vector<int32> private 이 노드가 의존하는 노드 ID 목록
m_dependents vector<int32> private 이 노드에 의존하는 노드 ID 목록
m_unfinishedDependencies Atomic<int32> private 미완료 의존성 카운터

Public API

TaskNode* DependsOn(TaskNode* other)
  • 이 노드가 other에 의존함을 선언 (other 완료 후 실행)
  • 양방향 등록: this->m_dependencies에 other.id 추가, other->m_dependents에 this.id 추가
  • this 반환 (체이닝 가능)
int32 GetId() const
  • 노드 ID 반환
const std::string& GetName() const
  • 노드 이름 반환
4.3 TaskGraph — DAG 실행기

멤버 변수

멤버 타입 설명
m_nodes vector<unique_ptr<TaskNode>> 모든 노드 소유
m_nextId int32 다음 할당할 노드 ID

Public API 상세

TaskNode* AddTask(const std::string& name, CallbackType callback)
  • 새 TaskNode 생성 (id 자동 할당)
  • m_nodes에 unique_ptr로 소유
  • 반환된 포인터로 DependsOn 체이닝 가능
  • 내부: new TaskNode(id, name, callback) + unique_ptr 래핑 (private 생성자이므로 make_unique 사용 불가)
void DependsOn(int32 taskId, int32 dependencyId)
  • ID 기반 의존성 선언 (TaskNode::DependsOn의 ID 버전)
  • 양쪽 노드에 ASSERT_TRIG_CRASH null 검증
void Execute()
  • 빈 그래프면 즉시 반환
  • Validate()로 사이클 검증 (실패 시 크래시)
  • 각 노드의 m_unfinishedDependencies를 m_dependencies.size()로 초기화
  • CountdownEvent(totalCount) 생성
  • 루트 노드(의존성 0개)를 SubmitReady()로 제출
  • countdown.Wait() — 모든 노드 완료까지 블로킹
void Clear()
  • 모든 노드 제거, m_nextId 초기화. 그래프 재사용 가능
bool Validate() const
  • DFS 기반 사이클 탐지
  • 상태 배열: 0=미방문, 1=방문중(스택), 2=완료
  • 방문중 노드를 재방문하면 사이클 → return false
  • 사이클 없으면 return true

Private 메서드

TaskNode* FindNode(int32 id) const
  • m_nodes에서 id로 선형 탐색
void SubmitReady(TaskNode* node, CountdownEvent* countdown)
  • 새 shared_ptr<JobQueue> 생성
  • DoAsync(pushOnly=true, ...) 호출 — 글로벌 워커 풀에 자동 등록
  • 람다 내용:
    1. node->m_callback() 실행
    2. node->m_dependents 순회: 각 dependent의 m_unfinishedDependencies 원자적 감소
    3. remaining이 0이면 해당 dependent도 SubmitReady() 호출
    4. countdown->Signal() — 이 노드 완료 신호
4.4 Execute() 내부 동작 흐름
Execute() 실행 순서 — A, B 병렬 → C 순차
Execute() 호출
     │
     ├─ Validate() — 사이클 검증
     │
     ├─ 각 노드 m_unfinishedDependencies 초기화
     │
     ├─ CountdownEvent(총 노드 수) 생성
     │
     ├─ 루트 노드(deps=0) 검색 → SubmitReady()
     │     │
     │     ├─ 노드 A (deps=0) → JobQueue → 워커 스레드
     │     └─ 노드 B (deps=0) → JobQueue → 워커 스레드
     │
     │         워커에서 A 완료:
     │           ├─ A의 dependents 순회
     │           ├─ C.unfinishedDeps: 2→1 (아직 >0)
     │           └─ countdown.Signal()
     │
     │         워커에서 B 완료:
     │           ├─ B의 dependents 순회
     │           ├─ C.unfinishedDeps: 1→0 → SubmitReady(C)
     │           └─ countdown.Signal()
     │
     │         워커에서 C 실행 → 완료:
     │           └─ countdown.Signal()
     │
     └─ countdown.Wait() → 모든 Signal 수신 후 반환
4.5 사용 예시 
// 물리 파이프라인 의존성
TaskGraph graph;

auto* broadPhase  = graph.AddTask("BroadPhase", [&]()
{
    physics.BroadPhase();
});

auto* narrowPhase = graph.AddTask("NarrowPhase", [&]()
{
    physics.NarrowPhase();
});

auto* resolve = graph.AddTask("Resolve", [&]()
{
    physics.ResolveContacts();
});

// Resolve는 BroadPhase와 NarrowPhase 둘 다 끝나야 실행
resolve->DependsOn(broadPhase);
resolve->DependsOn(narrowPhase);

graph.Execute();  // BroadPhase, NarrowPhase 병렬 → Resolve 순차

// 그래프 재사용
graph.Clear();

// ID 기반 의존성 (포인터 대신)
auto* a = graph.AddTask("A", []() { /*...*/ });
auto* b = graph.AddTask("B", []() { /*...*/ });
graph.DependsOn(b->GetId(), a->GetId());  // B depends on A
graph.Execute();

// 사이클 검증
auto* x = graph.AddTask("X", []() {});
auto* y = graph.AddTask("Y", []() {});
x->DependsOn(y);
y->DependsOn(x);  // 순환!
bool valid = graph.Validate();  // false 반환
SECTION 05

ThreadSafeHashMap<K, V>

5.1 클래스 개요

헤더 전용 템플릿. 기존 Lock (RW 스핀락)으로 std::unordered_map을 감싼 스레드 안전 해시맵입니다.

설계 원칙:

  • ▸ 읽기 연산: READ_LOCK (다수 동시 읽기 허용)
  • ▸ 쓰기 연산: WRITE_LOCK (단독 쓰기)
  • ▸ 기존 ResourceCache<T> 패턴의 수동 Lock + unordered_map 조합을 범용화
5.2 멤버 변수
멤버 타입 설명
m_locks[1] Lock USE_LOCK 매크로. RW 스핀락
m_map std::unordered_map<K, V> 내부 해시맵
5.3 읽기 API (READ_LOCK)
bool Find(const K& key, V& outValue) const
  • key에 해당하는 값을 outValue에 복사
  • 존재하면 true, 없으면 false
  • 스레드 안전: READ_LOCK (다수 동시 호출 가능)
V* FindUnsafe(const K& key)
  • 내부 값에 대한 포인터 직접 반환
  • 락 없음 — 호출자가 외부 락을 보유해야 함
  • 성능 임계 경로에서 복사 오버헤드 제거용
bool Contains(const K& key) const
  • key 존재 여부만 확인
  • READ_LOCK 하에 find != end 검사
size_t Size() const
  • 현재 원소 수 반환 (READ_LOCK)
bool Empty() const
  • 비어 있는지 확인 (READ_LOCK)
void ForEachRead(std::function<void(const K&, const V&)> func) const
  • READ_LOCK 하에 전체 순회
  • 콜백에 const 참조 전달 (읽기 전용)
5.4 쓰기 API (WRITE_LOCK)
bool Insert(const K& key, const V& value)
  • key가 없을 때만 삽입 (emplace)
  • 삽입했으면 true, 이미 존재하면 false
bool InsertOrAssign(const K& key, V value)
  • key가 없으면 삽입, 있으면 덮어씀
  • 항상 true 반환
V& operator[](const K& key)
  • std::unordered_map과 동일 의미
  • key가 없으면 기본값으로 생성
  • WRITE_LOCK 하에 참조 반환
bool Erase(const K& key)
  • key 제거. 제거했으면 true, 없으면 false
void Clear()
  • 전체 삭제 (WRITE_LOCK)
void ForEach(std::function<void(const K&, V&)> func)
  • WRITE_LOCK 하에 전체 순회
  • 콜백에 non-const 참조 전달 (수정 가능)
5.5 사용 예시 
// 텍스처 캐시
ThreadSafeHashMap<std::string, Texture*> textureCache;

// 삽입 (워커 스레드에서 안전)
textureCache.Insert("albedo", loadedTexture);

// 조회 (메인 스레드에서 안전)
Texture* tex = nullptr;
if (textureCache.Find("albedo", tex))
{
    material->SetTexture("AlbedoMap", tex);
}

// 존재 확인
if (textureCache.Contains("normal"))
    // ...

// 덮어쓰기
textureCache.InsertOrAssign("albedo", newTexture);

// 순회
textureCache.ForEachRead([](const std::string& key, Texture* const& tex)
{
    LogTexture(key, tex);
});

// 제거
textureCache.Erase("albedo");
textureCache.Clear();  // 전체 삭제
SECTION 06

의존성 관계

PHASE 1 (기존) PHASE 2 (신규)
CountdownEvent ◄────────── TaskGraph (Execute 내부에서 CountdownEvent 사용)
JobQueue ◄────────── TaskGraph (SubmitReady 내부에서 JobQueue 사용)
ParallelFor SpinWait 독립 모듈
Future / Promise EventSignal 독립 모듈
Lock / USE_LOCK ◄────────── ThreadSafeHashMap 기존 Lock 시스템 활용

TaskGraph는 Phase 1의 CountdownEvent와 기존 JobQueue에 의존합니다. SpinWait, EventSignal, ThreadSafeHashMap은 독립 모듈입니다. ThreadSafeHashMap은 기존 Lock 시스템(USE_LOCK / READ_LOCK / WRITE_LOCK)을 활용합니다.

SECTION 07

주의사항 및 제약

⚠ SpinWait는 인스턴스별 사용
여러 스레드가 하나의 SpinWait을 공유하면 안 됩니다. 스레드 로컬 변수 또는 스택 변수로 생성하세요. 공유할 경우 m_count 경쟁으로 백오프 단계가 올바르게 작동하지 않습니다.
⚠ EventSignal은 Auto-reset
Set() 호출 시 대기자 1명만 깨우고 자동으로 비신호 상태로 복귀합니다. 여러 대기자를 동시에 깨우려면 CountdownEvent를 사용하세요.
⚠ TaskGraph::Execute()는 블로킹
모든 노드 완료까지 호출 스레드가 대기합니다. 메인 스레드에서 호출 시 프레임 멈춤에 주의하세요. 프레임 내 실행 시에는 비동기 Execute 래퍼를 별도 구성하거나 워커 스레드에서 호출하세요.
⚠ TaskGraph 사이클 금지
DependsOn으로 순환 의존성을 만들면 Execute() 시 ASSERT_TRIG_CRASH가 발생합니다. Validate()로 사전 검증이 가능합니다.
⚠ TaskGraph는 Execute() 중 수정 금지
Execute() 호출 중에 AddTask()나 DependsOn()을 호출하면 정의되지 않은 동작입니다. 모든 노드를 등록한 후 Execute()를 호출하세요.
⚠ ThreadSafeHashMap::FindUnsafe()는 락 없음
호출자가 외부에서 락을 보유해야 합니다. 잘못 사용하면 데이터 레이스가 발생합니다. 성능 임계 경로 외에는 Find()를 사용하세요.
⚠ ThreadSafeHashMap::operator[]는 WRITE_LOCK
읽기 목적으로도 WRITE_LOCK을 획득합니다. 읽기 전용이면 반드시 Find()를 사용하세요. 불필요한 쓰기 락 경쟁을 방지해 성능을 개선합니다.
